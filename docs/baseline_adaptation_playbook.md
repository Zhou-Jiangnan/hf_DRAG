# 当外部方法和当前代码不适配时：基线对比落地手册

## 1) 先回答你的核心担心

是的，**只和自己方法对比说服力通常不够**。

但在工程不兼容时，不必强行“完全复现对方系统”，可以采用分层对标策略：

- **L1 可直接复用基线**（你代码里原生有）
- **L2 机制等价基线**（实现关键机制，不复现全部细节）
- **L3 论文结果对照**（同任务/同指标的外部结果用于定性定位）

这样评审一般会接受，前提是你把“适配边界和公平性”写清楚。

---

## 2) 你项目可立即执行的“强说服力”基线组

### G1（原生可跑，必须）

1. Centralized-PPO
2. FedAvg-PPO
3. FedAvg-PPO + DP
4. FedAvg-PPO + DP + MIA

### G2（机制等价，建议新增）

5. FedAvg-PPO + Robust Aggregation（median 或 trimmed mean）
6. FedAvg-PPO + Poisoning（sign-flip）
7. FedAvg-PPO + Poisoning + 防御（DP 或 RobustAgg）

> G2 不依赖外部大框架，改动只在聚合与更新注入环节，和你当前代码最适配。

### G3（外部论文对照，写作层）

- 对照 FedRAG / FedE4RAG / FedMosaic 的“思路与效果区间”，
- 明确说明：由于系统架构差异，本工作采用机制等价对比而非端到端复现。

---

## 3) 不适配时的操作步骤（实操）

### Step 1：冻结统一实验协议（先做）

- 同数据集、同 seeds、同训练预算（轮数×本地epoch）
- 同指标：EM/F1、MIA-AUC、Attack Advantage、通信估算

### Step 2：先做机制映射，不做系统复刻

- 外部方法若强调“鲁棒聚合”→你实现 median/trimmed-mean
- 外部方法若强调“参数高效通信”→你实现只同步子模块参数（可选）
- 外部方法若强调“防投毒”→你实现 sign-flip + defense 组合

### Step 3：写清公平性与限制

论文实验节专门加一段：

- 哪些基线是“原生实现”
- 哪些是“机制等价实现”
- 哪些是“引用对照（未复现）”

### Step 4：给评审“可信度证据”

- 3 seeds + mean±std
- 关键图：Pareto（MIA-AUC vs F1）
- 攻击前后对照图（Poison -> Defense）

---

## 4) 两周执行计划（最稳妥）

### Week 1

- D1：统一协议与脚本（seed、预算、日志字段）
- D2-D3：跑 G1
- D4-D5：实现并跑 G2(robust agg)

### Week 2

- D1-D2：实现并跑 G2(poison + defense)
- D3：汇总表格、误差条
- D4：写“公平性与限制”
- D5：组会彩排/论文初稿

---

## 5) 一句话策略

**不要被“外部方法不兼容”卡住。先用机制等价对比把故事闭环，再用文献对照补外部可信度。**
