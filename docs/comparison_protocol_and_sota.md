# 外部算法迁移到本模型后的对比：怎么判定效果？是否必须 SOTA？

## 1) 先回答两个关键问题

### Q1. 把别人的算法放到你模型上跑，效果怎么算？

要按“**统一协议 + 机制等价**”来算，不是只看绝对分数：

- **统一协议**：同数据、同seed、同训练预算、同评测脚本；
- **机制等价**：复现对方方法的核心机制（如 robust aggregation / DP / poisoning defense），
  不要求端到端完全同架构。

最终比较的是：

1. 任务性能（F1/EM/hops/messages）
2. 隐私指标（MIA-AUC / Attack Advantage）
3. 鲁棒指标（Poison 场景下降幅）
4. 效率指标（时间/通信）

### Q2. 必须在你模型上达到 SOTA 才能发吗？

**不必须。** 对于系统安全/隐私类工作，评审更看重：

- 问题是否真实且重要；
- 对比是否公平；
- 结论是否稳定（多seed、误差条）；
- 是否给出清晰 trade-off（隐私 vs 性能 vs 鲁棒）。

如果你能在“同等预算下”证明更好 trade-off，就有发表价值，即使不是绝对 SOTA。

## 2) 推荐的效果计算方式（可直接执行）

## A. 统一预算

- 训练预算：`rounds * local_epochs` 固定
- 采样比例：`fed_client_fraction` 固定
- 同一随机种子集合：`{0,1,2}`

## B. 统一输出表

每个方法都输出以下字段（均值±标准差）：

- `f1`, `exact_match`
- `mia_auc`, `mia_attack_advantage`
- `avg_num_hops`, `avg_num_messages`
- 训练耗时（可选）

## 3) 论文里如何写“不是SOTA”也有说服力

建议用以下叙事：

1. 本文目标不是追求单一任务最高分，而是研究联邦路由中的隐私-鲁棒-性能三方平衡；
2. 在统一协议下，本文方法在隐私泄露指标（MIA）和鲁棒性上更优或更稳定；
3. 代价可控（性能下降在可接受范围）。

---

## 4) 最小可发表结论模板

- 与 FedAvg 相比，DP/防御策略将 MIA-AUC 从 X 降到 Y；
- 同时 F1 仅下降 Z（或在投毒下反而提升）；
- 在 poison 场景中，鲁棒聚合显著降低性能退化。

这类“可解释 trade-off”通常比“单点 SOTA”更适合你当前方向。

---

## 5) 实操建议（你下一步）

1. 先完成 6~7 组统一协议实验；
2. 补齐 3 seeds 与误差条；
3. 把外部方法按“机制等价”接入，不做全栈复刻；
4. 先追求结论闭环，再考虑冲更高分。
